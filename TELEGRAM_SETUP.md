# Настройка сбора данных из Telegram

Сбор идёт через библиотеку **Telethon**: нужны **API ID / API Hash** с сайта Telegram и **один раз** интерактивный вход (телефон + код). Сессия сохраняется в Docker-volume и дальше используется backend’ом.

---

## Шаг 1. Получить API ID и API Hash

1. Откройте в браузере: **https://my.telegram.org**
2. Войдите **номером телефона**, к которому привязан ваш Telegram (код придёт в Telegram или SMS).
3. После входа нажмите **«API development tools»** (или аналогичный пункт).
4. Если приложения ещё нет — **создайте** (можно произвольные короткие название/краткое имя).
5. Скопируйте:
   - **api_id** — целое число (например `12345678`);
   - **api_hash** — длинная строка из букв и цифр.

Эти данные **не** являются токеном бота: это ключи **пользовательского** API Telethon (как «ваше приложение» в экосистеме Telegram).

---

## Шаг 2. Прописать переменные в `.env`

В каталоге **`llm-rag`** (рядом с `docker-compose.yml`) отредактируйте файл **`.env`** (создайте из `.env.example`, если его нет):

```env
TELEGRAM_API_ID=ваш_числовой_api_id
TELEGRAM_API_HASH=ваш_api_hash_строкой
TELEGRAM_SESSION_NAME=llm_rag_session
```

- Без кавычек вокруг значений, без пробелов вокруг `=`.
- `TELEGRAM_SESSION_NAME` можно не менять — имя файла сессии на диске.

Опционально (если позже понадобится прокси — см. раздел «Проблемы с сетью»):

```env
TELEGRAM_PROXY=socks5://127.0.0.1:1080
```

Переменные подхватывает **Docker Compose** автоматически из этого `.env`.

---

## Шаг 3. Пересобрать backend и войти в Telegram один раз

Выполняйте команды из каталога **`llm-rag`**:

```bash
docker compose build backend
make telegram-login
```

Что произойдёт:

1. Соберётся образ backend (в том числе зависимости `telethon`, `PySocks`).
2. Запустится интерактивный скрипт входа: попросит **номер телефона**, затем **код из Telegram**, при необходимости **пароль двухфакторной аутентификации**.
3. Файл сессии сохранится в volume **`telegram_session`** (путь внутри контейнера: `/data/telegram_session`), его потом читает работающий backend.

Если увидите **«Вход выполнен»** — авторизация прошла успешно.

Запустите или перезапустите сервисы:

```bash
docker compose up -d backend
```

---

## Шаг 4. Добавить источник Telegram в интерфейсе

1. Откройте приложение (например **http://localhost:3000**), войдите в аккаунт.
2. Выберите **проект**.
3. В боковой панели вкладка **«Источники»**.
4. Вкладка **«Telegram»**:
   - **Название** — любое человекочитаемое (для списка).
   - Поле чата / канала — укажите **один из вариантов**:
     - публичный канал: **`@username`** (например `@durov`);
     - или ссылку **`https://t.me/username`**;
     - для групп/чатов часто нужен **числовой ID** (например `-1001234567890`), его можно узнать через ботов типа `@userinfobot` / `@getidsbot` или через клиент Telegram (для приватных каналов без `@` это основной способ).
5. Нажмите **«Добавить источник»**.
6. Нажмите **«Обновить базу знаний»** в шапке чата — пойдёт загрузка сообщений за последние дни (лимиты задаются в `.env`: `TELEGRAM_MESSAGE_LIMIT`, `TELEGRAM_DAYS_BACK`).

У аккаунта Telegram, под которым вы вошли через `make telegram-login`, должна быть **возможность читать** этот канал/чат (для закрытых каналов — предварительно вступите в канал из того же аккаунта).

## Проблемы с сетью («Connection to Telegram failed»)

Telegram использует отдельный протокол (MTProto). Если из вашей сети или из Docker **нет исходящего доступа** к серверам Telegram, Telethon выдаст ошибки подключения.

**Что сделать:**

1. На **хосте** (Linux/macOS/Windows) включите **VPN** или другой работающий способ доступа к Telegram.
2. Если VPN даёт **локальный SOCKS5** (часто `127.0.0.1` и порт `1080` или см. настройки клиента), добавьте в **`.env`**:

   ```env
   TELEGRAM_PROXY=socks5://127.0.0.1:1080
   ```

   У backend в `docker-compose` включён **`network_mode: host`**, поэтому **`127.0.0.1` на хосте** доступен процессу в контейнере как тот же localhost — прокси на машине можно так и указать.

3. Пересоберите backend и снова выполните **`make telegram-login`** и **`docker compose up -d backend`**.

При необходимости увеличьте ожидание:

```env
TELEGRAM_CONNECTION_RETRIES=20
TELEGRAM_TIMEOUT_SEC=30
```

## Краткий чеклист

| Действие | Где |
|----------|-----|
| API ID / Hash | https://my.telegram.org → API development tools |
| Ключи в `.env` | каталог `llm-rag/.env` |
| Первый вход Telethon | `make telegram-login` из `llm-rag` |
| Источник в UI | Источники → Telegram → канал `@...` или ID |
| Индексация | Кнопка «Обновить базу знаний» |
| Прокси при блокировках | `TELEGRAM_PROXY=socks5://127.0.0.1:1080` в `.env` |

## Полезные команды

```bash
# Логи backend (ошибки Telethon видны здесь)
docker compose logs -f backend

# Только пересборка и рестарт backend
docker compose build backend && docker compose up -d backend

# Проверка доступа к Telegram тем же способом, что и приложение (Telethon + прокси из .env)
make telegram-check
```

### Быстрая проверка «есть ли интернет до Telegram»

Это **не** то же самое, что MTProto (как у клиента Telegram), но показывает, что HTTPS до `api.telegram.org` доходит:

```bash
curl -sS -o /dev/null -w "%{http_code}\n" https://api.telegram.org
```

Если долго висит или ошибка — сеть режет доступ; для MTProto используйте **`make telegram-check`** или **`make telegram-login`**.

Если сессия «сбилась», можно снова выполнить `make telegram-login` (тот же номер телефона и аккаунт, что и раньше, либо новая сессия с другим `TELEGRAM_SESSION_NAME` в `.env` и повторный вход).
